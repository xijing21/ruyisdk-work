# RuyiSDK 用户注册管理

## 背景
为了满足RuyiSDK社区推广和用户统计的需求，针对RuyiSDK产品（包括包管理器、IDE、官网、技术论坛、Github代码仓库、微信公众号RUYISDK）进行用户注册/登录管理，以 **引导** 为主，而非 **强制** ，引导用户登录，积累RuyiSDK用户量，作为后续RuyiSDK社区规模衡量参考之一。

## RuyiSDK 引导注册方案

- 在技术论坛Discourse上启用并配置OAuth服务端功能，使Discourse能够作为一个OAuth 2.0提供商，接受来自IDE、官网等客户端的授权请求。在IDE插件和官网中加入引导用户注册的入口。
- github 作为 ruyisdk 的代码托管平台，目前托管了 40+ 个 ruyisdk 的仓库；后续可以考虑通过 github 授权的方式登录ruyisdk。统一配置在Discourse后台，由Discourse作为“网关”接入。
- 微信授权：RUYISDK 公众号目前也积累了一些用户，作为目前的主流讯息入口之一，后续可以考虑通过 微信授权方式登录ruyisdk，在中国地区这种授权登录方式将很有优势。（芯来科技采用的这一方式）
- 用户数据：只存在Discourse的数据库里。且只采集最小用户数据：邮箱和密码（或者github/微信授权信息）

这样做的好处是：
- 不改现有论坛：论坛老用户不受影响，直接用原有账号密码登录所有新地方。
- 一次接入，全网通用：只需开发一次Discourse的OAuth客户端，所有产品都复用。
- 降低合规风险：你依然只维护Discourse那一套用户数据，没有新增数据存储点。

### IDE插件注册埋点
问题：IDE不注册也能用，如何引导用户注册？如何让注册成为"增值"，而不是"门槛"。

1. 打开/启动 RuyiSDK插件时：首次启动时弹窗，在介绍RuyiSDK的同时，引导用户注册。
  - 用户完成注册，记住注册信息，以后启动时默认保持登录状态。
  - 用户直接关闭跳过注册的，后续在必要的“增值”服务中埋点。

2. 增值服务埋点的场景
  - （1）技术论坛增值服务：
    - 通过诸如webview的技术内嵌ruyisdk论坛（已经登录的默认直接登录，没有登录的使用论坛默认的发帖/回复跳转登录）
    - 针对已注册登录的用户提供：论坛热门帖子和新闻
    - 论坛关键词搜索与内容展示：如用户当前设备是milkvduo，通过论坛搜索功能展示所有milkvduo有关的论坛帖子
    - 提问：支持用户提问，并将提问直接提交到ruyisdk论坛
  - （2）AI增值服务
    - 让AI学习ruyisdk的文档和技术社区的内容，提供开发帮助（具体有待调研和定义）
  - （3）其它用户个性化的配置同步、云端备份等需求（待挖掘）

### 官网注册埋点
问题：注册功能加在哪里？什么时候跳出来？

1. 在网站的右上角加上用户图标：如果登录过则记住用户信息，下次自动登录。如果没有注册登录过，点击图标则调到注册/登录页面；如 https://www.openeuler.openatom.cn/zh/ 右上角处
2. 场景化埋点：待挖掘和补充


## 问题
1. Discourse 只维护注册用户数据，如果涉及到“用户个性化的配置同步、云端备份等需求”，那么这个用户个性化的配置和云备份的数据，存储在哪里？
  推荐：ruyi后端存储独立业务数据库 + 用户ID关联  
  数据流转流程：
      - 用户登录：在IDE/CLI中通过Discourse OAuth登录，获取user_id
      - 读写业务数据：IDE/CLI带着user_id访问ruyi后端的API
      - 数据同步：ruyi后端将配置/历史等数据存入自己的数据库
      - 跨设备拉取：用户在新设备登录时，IDE/CLI再次携带user_id请求ruyi后端，拉取云端数据

2. 包管理器是一个CLI的工具，是否需要进行引导注册？
  暂无用户信息接入需求。

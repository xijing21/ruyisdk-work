## RuyiSDK 实习生入门指南（记录与汇报篇）

> 本文档是 RuyiSDK 实习生入门指南的一部分，帮助你建立“随时记录”的工作习惯，将日常产出转化为可见的成果，并规范地进行周报/月报同步。

欢迎加入 RuyiSDK！这篇指南会帮助你了解如何在实习期间高效地记录和展示你的工作，让每一份努力都变得清晰、可追溯、可复用。

### 一、养成随时记录的习惯

在远程协作中，信息容易碎片化。你可能今天修复了一个编译错误，明天跑通了一块开发板，但如果不及时记录，几天后细节就会模糊，更不用说向 mentor 清晰地展示你的贡献了。

**养成随时记录的习惯，能带来三个直接的好处：**

1. **积累自己的“作品集”**  
   把你做过的任务（比如适配某开发板、解决某编译问题）整理成一个独立的公开仓库，里面放上代码、脚本、踩坑记录。这个仓库就是你能力的直接证明——无论是面试、申请学校，还是分享给朋友，你都可以自信地丢出一个链接。多个这样的仓库，会让你的 GitHub 主页变得充实而有说服力。

2. **符合“可见交付”的要求**  
   RuyiSDK 非常看重可见交付（Visible Deliverables）：你做的东西必须能被看到、被用到、被验证。公开仓库中的代码、文档 PR、论坛帖子、技术分享视频等，都是可见产出。这些链接将成为你周报/月报的核心内容，也是 mentor 评估你工作的重要依据。

3. **用输出倒逼深度理解（费曼学习法）**  
   费曼学习法告诉我们：**真正掌握一个知识，最好的方式就是把它写出来、讲给别人听**。当你把解决问题的过程写成文档、录制成视频、或者在论坛上发帖分享时，你其实是在用自己的语言重新组织思路——这往往会让你发现自己理解中的盲点，从而加深对技术的掌握。  
   我们鼓励你不仅写下来，还可以**在团队内部做一次技术分享**（比如一次 15 分钟的线上分享会），把学到的知识讲出来。文字记录 + 口头表达，形成一个完整的学习闭环，效果远超“闷头搞”。

> 在 RuyiSDK，推荐的完整工作流程为：  
> **完成任务 → 撰写文档/发个帖子 → [可选]技术分享/录制视频 → 发到群里（一句简短介绍+链接） → 把链接贴进周报/月报**

这个流程把你零散的工作串联成了一条“可追溯、可展示”的链条。其中，前四步都是在“产出可见内容”，最后一步则是把这些内容汇集起来，形成你远程协作的阶段性总结。**周报和月报正是这条链条的终点**：它们既是 mentor 了解你进度的窗口，也是每月劳务费评定的主要依据。没有它们，你的所有努力就缺少了一次系统性的“归档”，mentor 也无法高效地评估你的产出。

所以，请从现在开始：每完成一个小任务，就随手创建一个公开仓库、写一篇文档、发一条论坛帖子——然后把这些链接收集起来。这样，到了写周报/月报的时候，你就不再是“绞尽脑汁回忆”，而是“轻松地复制粘贴链接”。

### 二、核心理念：把你的工作“晒”出来

在 RuyiSDK，我们看重“可见交付”。简单说，就是你做的东西能被看到、被用到、被验证。

上面的工作流程已经很清晰地展示了这一点：从完成任务到最终贴进周报，每一步都在让产出变得“可见”。

当然，也有例外：如果 mentor 安排了**不适合公开的任务**（比如内部调研、未公开代码库的预研），那就不用勉强公开，直接和 mentor 沟通就好。

### 三、你的两种记录仓库

为了把“随时记录”落地，你需要在 GitHub/Gitee 上维护两种仓库：**任务记录公开仓库**（按项目/任务独立创建）和 **周报/月报仓库**（用于汇总同步）。

#### 3.1 任务记录公开仓库（建议按任务独立创建）

这类仓库是针对具体任务创建的独立 public 仓库。它们是你“作品集”的核心：

- **什么时候创建？** 当你接到一个有一定复杂度的任务（比如适配一块新开发板、解决一个典型问题、开发一个小工具）时，就可以新建一个独立仓库。
- **里面放什么？**  
  - 代码/脚本（完整可运行）  
  - README 说明（如何运行、依赖、效果）  
  - 踩坑记录 / 常见问题  
  - 示例输出或截图  
- **起名建议**：`<任务关键词>-demo` 或 `<板子名>-notes`，例如 `licheepi-4a-demo`、`ruyisdk-tips`、`duo-buildroot-fix`。
- **好处**：  
  - 每个仓库都是一个独立的“作品”，方便日后复用和分享。  
  - 别人可以直接通过你的仓库了解你的能力。  
  - 这些仓库的链接可以贴进你的周报/月报，成为可见产出的重要组成部分。

> 至于是**一个任务一个仓库**  还是 **一个仓库通过不同二级目录管理不同任务** ，这个由你自己衡量，没有绝对的标准。

#### 3.2 周报/月报仓库（用于汇总同步）

除了上述任务仓库，你还需要一个专门的 public 仓库来存放周报和月报（名字自取，如 `ruyisdk-weekly`）。它的作用是：

- **每周一次**：写一篇周报，把当周完成的所有任务对应的链接（包括 PR、文档、论坛帖子、任务公开仓库等）贴进去。
- **每月一次**：写一篇月报，汇总本月所有链接，作为月度劳务费评定的依据。

**目录结构参考**：

```
ruyisdk-weekly/
├── 202604/
│   ├── week1.md
│   ├── week2.md
│   ├── week3.md
│   ├── week4.md
│   └── month-summary.md
├── 202605/
│   ├── week1.md
│   └── ...
```

**重点是产出周报和月报**，文件名和目录可以按你自己顺手的方式来。

**周报怎么写？（推荐按任务划分）**

每周新建一个周报文档，写清楚你做了什么，并把相关的公开链接贴出来。

##### 一个例子

```markdown
# 第1周周报（20260401 ~ 20260407）

## 任务一：在 LicheePi 4A 上跑通 RVV 向量化示例

### 相关产出（可见链接）
- 任务公开仓库：https://github.com/yourname/licheepi-4a-demo
- PR：修复 RVV 编译错误 → https://github.com/ruyisdk/ruyisdk-sdk/pull/123
- 文档：《LicheePi 4A RVV 使用指南》 → https://github.com/ruyisdk/docs/pull/89
- 论坛分享帖：https://ruyisdk.cn/forum/topic/123

### 本周进展
已完成代码提交和文档初稿，PR 在 review 中。

---

## 任务二：解决 duo-buildroot-sdk 的编译失败问题

### 相关产出（可见链接）
- 上报 Issue：https://github.com/ruyisdk/duo-buildroot-sdk/issues/45
- 论坛回复（临时解决方案）：https://ruyisdk.cn/forum/topic/122?reply=3

### 本周进展
问题已定位，等待 maintainer 确认。

---
> **小贴士**：如果某个任务不适合公开（mentor 明确说过不用公开），那就不需要写在这里，直接和 mentor 私下沟通进度就行。
```

几点说明：
- **可见产出**：一定要有真实可点击的 URL，不要只写“修复了一个 bug”而没链接。
- **不公开任务**：听 mentor 的，不用硬要“晒出来”。直接和 mentor 在群里或私聊沟通。

**提交时间**：每周一之前，提交上一周的周报。

**月报怎么写？（按任务汇总）**

每月底新建一个月总结文档，把本月所有可见产出的链接按任务分类汇总。

##### 例子

```markdown
# 2026年4月月度总结

## 任务一：在 LicheePi 4A 上跑通 RVV 向量化示例

### 完成情况
- ✅ PR #123 已合并
- ✅ 文档 PR #89 已合并
- ✅ 论坛分享帖已发布，获得一些点赞

### 所有产出链接汇总
- https://github.com/yourname/licheepi-4a-demo
- https://github.com/ruyisdk/ruyisdk-sdk/pull/123
- https://github.com/ruyisdk/docs/pull/89
- https://ruyisdk.cn/forum/topic/123

---

## 任务二：解决 duo-buildroot-sdk 编译失败问题

### 完成情况
- ✅ Issue #45 已被接受，修复 PR 由他人提交（我提供了复现步骤）
- ✅ 论坛回复被认为有帮助

### 所有产出链接汇总
- https://github.com/ruyisdk/duo-buildroot-sdk/issues/45
- https://ruyisdk.cn/forum/topic/122?reply=3

---

## 社区互动（非运营岗位）

### 本月互动记录
- 论坛有效回复：5条，链接见下方
- 微信群帮助别人：3次

> 非社区运营岗位的同学，如果这方面做得比较突出，mentor 会给一点小小的鼓励（不是硬性考核，就是一份认可）。

---

## 本月核心总结
完成2个开发任务，累计提交2个PR、1个Issue、多篇论坛回复。
```

> **注意**：月报里最好把链接再列一遍，不要只写“详见各周报”，这样 mentor 一眼就能看到全貌。

**提交时间**：每月最后一天之前，提交本月月报。月报是月度劳务费评定的主要依据，请认真梳理本月的所有可见产出。

**关于跨月任务的说明**：如果某任务在月末（比如4月30日）尚未完成，其产出难以在本月衡量，可以将其计入下一个月。原则是：**每个产出的劳务费只支付一次**，具体放在哪个月由你和 mentor 灵活协商即可。

### 四、哪些算“可见产出”？一些真实例子

| 类型 | 示例链接（格式参考） |
|------|----------------------|
| 任务公开仓库 | `https://github.com/yourname/licheepi-4a-demo` |
| 代码 PR | `https://github.com/ruyisdk/ruyisdk-sdk/pull/123` |
| 文档 PR | `https://github.com/ruyisdk/docs/pull/89` |
| Issue 反馈 | `https://github.com/ruyisdk/duo-buildroot-sdk/issues/45` |
| 论坛技术分享帖 | `https://ruyisdk.cn/forum/topic/123` |
| 论坛资讯帖 | `https://ruyisdk.cn/forum/topic/124` |
| 论坛回复 | `https://ruyisdk.cn/forum/topic/122?reply=3` |
| 自制视频教程 | 比如 B 站链接：`https://www.bilibili.com/video/BV1xx411c7mY` |
| 微信群热心解答 | 可以写“本周在群内帮助3位用户，附简要说明” |

**核心原则**：有公开链接就贴链接；mentor 说不公开的任务，就不用贴。

### 五、实习过程中的几个自然节点

在实习期间，你可能会经历几个阶段，提前了解会让过程更顺畅：

- **初始合同**：首次实习合同通常为 3 个月。这是一个互相了解、磨合的阶段。
- **阶段性回顾**：每 3 个月左右，mentor 会和你一起回顾之前的工作，看看完成了哪些成果，以及接下来希望做什么方向。这不是“考试”，而是帮助你明确成长路径。
- **续约考量**：如果双方都有意愿继续合作，合同可以续签。续签时，之前积累的周报、月报和你建立的公开仓库，就是最直观的能力证明。
- **后续规划**：每个阶段结束时，mentor 会和你一起商量下一阶段的大致目标和任务。你不需要自己独自规划，但需要了解并配合这个节奏，让工作更有连续性。

这些节点不是为了增加压力，而是为了让实习更有方向感，也让你每次回顾时都能看到自己的进步。

### 六、其他小建议

**多写文档，受益的是自己**  
写完代码顺手写个使用说明、踩坑记录，这些文档就是你以后面试时能拿出的“作品”。而且写出来之后，你往往会对那个问题理解得更深。

**AI 可以帮忙，但别忘了自己思考**  
用 AI 整理文字、生成草稿没问题，但要自己验证内容对不对，不要无脑复制。

**Git 习惯**：提交 PR 时记得用 `git commit -s` 加上 DCO 签名。

**论坛多转转**：不是运营岗也没关系，逛逛论坛、回答别人问题，既能学到东西，也能让 mentor 看到你的热情。

**不公开的任务**：按 mentor 说的做就行，不用纠结。直接和 mentor 沟通进展，他们会评估。

### 七、常见问题

**Q：这周只调研学习了，没有 PR 和 Issue，怎么办？**  
A：把调研笔记、学习心得整理成一篇**论坛帖子**或者 **GitHub Gist**，链接贴进周报——这就是很好的可见产出。如果 mentor 要求不公开，那就只写“完成内部调研，已发 mentor”。

**Q：PR 还没合并，可以写进周报吗？**  
A：当然可以。写“PR #123（review 中）”，链接照样贴。合并了再更新状态。

**Q：周报文件名必须叫 week1.md 吗？**  
A：不用，你觉得好认就行。比如 `2026-04-01_week1.md` 也可以。

**Q：月报里一定要重复贴链接吗？**  
A：是的，这样 mentor 不用翻前面的周报，能省时间。

**Q：内部分享（没有公开链接）算不算产出？**  
A：算有效产出！只是不属于“可见产出”。如果是 mentor 安排的不公开任务，直接和 mentor 确认就好。

**Q：跨月任务怎么处理比较方便？**  
A：如果任务在月底没有产生明确的可见产出（比如代码还在本地、文档没写完），可以等到下个月产出完成后，统一算在下个月的月报里。和 mentor 说一声就行，原则就是每个产出只领一次劳务费。

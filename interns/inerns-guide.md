# RuyiSDK 实习生入门指南

欢迎加入 RuyiSDK！这份指南会帮助你了解实习期间的一些工作习惯，特别是如何记录和展示你做的事情。希望它能让你更顺畅地融入团队，也能从这段实习中获得更多成长。


## 一、为自己而记：周报月报的意义

写周报、月报，是远程协作中一种合适的沟通方式，是每月底的一份成绩单，是向 mentor 展示你贡献的总结。同时，它也会帮助你养成记录和输出的习惯，成为你学习与成长的见证。

> **费曼学习法说**：真正掌握一个知识，最好的方式就是把它写出来、讲给别人听。  
> 你在周报里写下的每一条产出、每一段进展，都是在**用自己的话重新组织**你做过的实事。这个过程会帮你：
> - 发现思路中的盲点
> - 加深对技术细节的理解
> - 积累一份随时可以拿出来的“作品集”

想象一下：几个月后，当你找工作、申请学校、或者跟别人介绍自己做过什么时，你不需要绞尽脑汁回忆，只需要打开你的 GitHub 仓库——那里清清楚楚写着你在 RuyiSDK 完成的每个任务、每个 PR、每篇文档。**这才是最有说服力的简历。**

所以，这份指南不是要给你“加任务”，而是帮你养成一个**既能让 mentor 了解进度，又能让自己持续受益**的好习惯。

---

## 二、核心理念：把你的工作“晒”出来

在 RuyiSDK，我们很看重**可见交付（Visible Deliverables）**。简单说，就是你做的东西能被看到、被用到、被验证。

> 一个简单的流程：完成任务 → 写下文档/发个帖子 → 把链接贴进周报

**有没有产出，以你在公开渠道留下的痕迹为准。** 当然，也有例外：如果 mentor 安排了**不适合公开的任务**（比如内部调研、未公开代码库的预研），那就不用勉强公开，直接和 mentor 沟通就好。

---

## 三、仓库结构

建议在你的 **GitHub/Gitee** 上创建一个 **public 仓库**（名字你自己定，比如 `ruyisdk-work`），在里面放周报和月报。目录结构可以参考下面这样：

```
ruyisdk-work/
├── 2026-04/
│   ├── week1.md
│   ├── week2.md
│   ├── week3.md
│   ├── week4.md
│   └── month-summary.md
├── 2026-05/
│   ├── week1.md
│   └── ...
```

**重点是产出周报和月报**，文件名和目录可以按你自己顺手的方式来，上面的例子只是帮你理解。

> 友情提示：除了这个用来放周报月报的主仓库，你还可以根据实际需要，为某些项目或任务单独创建其他的 public 仓库。
> 
> 比如：你正在适配一块新开发板，可以单独建一个 licheepi-4a-demo 仓库，里面放你的测试代码、运行脚本、踩坑记录；或者你整理了一份常见问题集，可以建一个 ruyisdk-tips 仓库，把散落在周报里的经验集中起来。
> 
> 这样做的好处是：每个独立的仓库就像一个“作品”，方便你日后复用代码、分享给别人，也让你的 GitHub 主页看起来更充实。而且，这些仓库里的内容也可以作为可见产出的链接，贴进你的周报和月报里。

## 四、周报怎么写（推荐按任务划分）

每周新建一个周报文档，写清楚你做了什么，并把相关的公开链接贴出来。

### 一个例子

```markdown
# 第1周周报（2026-04-01 ~ 2026-04-07）

## 任务一：在 LicheePi 4A 上跑通 RVV 向量化示例

### 相关产出（可见链接）
- PR：修复 RVV 编译错误 → https://github.com/ruyisdk/ruyisdk-sdk/pull/123
- 文档：《LicheePi 4A RVV 使用指南》 → https://github.com/ruyisdk/docs/pull/89
- 论坛分享帖：https://ruyisdk.cn/forum/topic/123

### 本周进展
已完成代码提交和文档初稿，PR 在 review 中。

---

## 任务二：解决 duo-buildroot-sdk 的编译失败问题

### 相关产出（可见链接）
- 上报 Issue：https://github.com/ruyisdk/duo-buildroot-sdk/issues/45
- 论坛回复（临时解决方案）：https://ruyisdk.cn/forum/topic/122?reply=3

### 本周进展
问题已定位，等待 maintainer 确认。

---
> **小贴士**：如果某个任务不适合公开（mentor 明确说过不用公开），那就不需要写在这里，直接和 mentor 私下沟通进度就行。
```

几点说明：
- **可见产出**：一定要有真实可点击的 URL，不要只写“修复了一个 bug”而没链接。
- **不公开任务**：听 mentor 的，不用硬要“晒出来”。直接和 mentor 在群里或私聊沟通。

---

## 五、月报怎么写（按任务汇总）

每月底新建一个月总结文档，把本月所有可见产出的链接按任务分类汇总。如果社区互动比较活跃，也可以顺便提一下。

### 例子

```markdown
# 2026年4月月度总结

## 任务一：在 LicheePi 4A 上跑通 RVV 向量化示例

### 完成情况
- ✅ PR #123 已合并
- ✅ 文档 PR #89 已合并
- ✅ 论坛分享帖已发布，获得一些点赞

### 所有产出链接汇总
- https://github.com/ruyisdk/ruyisdk-sdk/pull/123
- https://github.com/ruyisdk/docs/pull/89
- https://ruyisdk.cn/forum/topic/123

---

## 任务二：解决 duo-buildroot-sdk 编译失败问题

### 完成情况
- ✅ Issue #45 已被接受，修复 PR 由他人提交（我提供了复现步骤）
- ✅ 论坛回复被认为有帮助

### 所有产出链接汇总
- https://github.com/ruyisdk/duo-buildroot-sdk/issues/45
- https://ruyisdk.cn/forum/topic/122?reply=3

---

## 社区互动（非运营岗位）

### 本月互动记录
- 论坛有效回复：5条，链接见下方
- 微信群帮助别人：3次

> 非社区运营岗位的同学，如果这方面做得比较突出，mentor 可能会给一点小小的鼓励（不是硬性考核，就是一份认可）。

---

## 本月核心总结
完成2个开发任务，累计提交2个PR、1个Issue、多篇论坛回复。
```

> 注意：月报里最好把链接再列一遍，不要只写“详见各周报”，这样 mentor 一眼就能看到全貌。

## 六、哪些算“可见产出”？一些真实例子

| 类型 | 示例链接（格式参考） |
|------|----------------------|
| 代码 PR | `https://github.com/ruyisdk/ruyisdk-sdk/pull/123` |
| 文档 PR | `https://github.com/ruyisdk/docs/pull/89` |
| Issue 反馈 | `https://github.com/ruyisdk/duo-buildroot-sdk/issues/45` |
| 论坛技术分享帖 | `https://ruyisdk.cn/forum/topic/123` |
| 论坛资讯帖 | `https://ruyisdk.cn/forum/topic/124` |
| 论坛回复 | `https://ruyisdk.cn/forum/topic/122?reply=3` |
| 自制视频教程 | 比如 B 站链接：`https://www.bilibili.com/video/BV1xx411c7mY` |
| 微信群热心解答 | 可以写“本周在群内帮助3位用户，附简要说明” |

**核心原则**：有公开链接就贴链接；mentor 说不公开的任务，就不用贴。

## 七、简单的时间要求

- **每周**：写周报，把本周的可见产出链接贴上。建议每周一之前完成上周的总结。
- **每月**：写月报，汇总本月所有链接。建议每月最后一天写完。
- **同步**：每次更新后，可以在实习生群里说一声“周报/月报已更新：链接”，方便 mentor 及时看到。

---

## 八、几个小建议

**多写文档，受益的是自己**  
写完代码顺手写个使用说明、踩坑记录，这些文档就是你以后面试时能拿出的“作品”。而且写出来之后，你往往会对那个问题理解得更深。

**AI 可以帮忙，但别忘了自己思考**  
用 AI 整理文字、生成草稿没问题，但要自己验证内容对不对，不要无脑复制。

**Git 习惯**：提交 PR 时记得用 `git commit -s` 加上 DCO 签名。

**论坛多转转**：不是运营岗也没关系，逛逛论坛、回答别人问题，既能学到东西，也能让 mentor 看到你的热情。

**不公开的任务**：按 mentor 说的做就行，不用纠结。直接和 mentor 沟通进展，他们会评估。

## 九、常见问题

**Q：这周只调研学习了，没有 PR 和 Issue，怎么办？**  
A：把调研笔记、学习心得整理成一篇**论坛帖子**或者 **GitHub Gist**，链接贴进周报——这就是很好的可见产出。如果 mentor 要求不公开，那就只写“完成内部调研，已发 mentor”。

**Q：PR 还没合并，可以写进周报吗？**  
A：当然可以。写“PR #123（review 中）”，链接照样贴。合并了再更新状态。

**Q：周报文件名必须叫 week1.md 吗？**  
A：不用，你觉得好认就行。比如 `2026-04-01_week1.md` 也可以。

**Q：月报里一定要重复贴链接吗？**  
A：是的，这样 mentor 不用翻前面的周报，能省时间。

**Q：内部分享（没有公开链接）算不算产出？**  
A：算有效产出！只是不属于“可见产出”。如果是 mentor 安排的不公开任务，直接和 mentor 确认就好。

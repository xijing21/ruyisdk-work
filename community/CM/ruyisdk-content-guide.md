
# RuyiSDK 内容建设者实操指南

本指南面向所有参与 RuyiSDK 内容建设的人员（研发、测试、运营、PM、实习生、社区贡献者）。

**请先阅读以下文档**：
1. 《RuyiSDK 内容建设规划书》（`CONTENT_STRATEGY.md`）— 了解整体架构和工作包
2. 《RuyiSDK 内容运营策略与规范》（`CONTENT_OPERATIONS.md`）— 了解内容分级、系列命名和推广规范

---

## 1. 内容类型与对应规范

| 内容类型 | 对应层次 | 用途 | 托管仓库/平台 | 主要内容 |
|----------|----------|------|---------------|----------|
| 官方文档（修正/新增） | 官方文档层 | 权威参考 | `ruyisdk/docs` | 安装指南、命令手册、配置说明 |
| 工具使用案例 | 案例/实战层 | 教某个工具怎么用 | `ruyisdk/board-docs` | 场景描述、步骤、预期输出 |
| 实战体验包（含文档） | 案例/实战层 | 完整场景复现 | `ruyisdk/board-docs` + `ruyisdk/packages-index` | 五要素体验包（见下文） |
| 入门指南（图文/视频） | 推广内容层 | 短平快引流 | 论坛、B站、知乎、掘金等 | 图文800字以内；视频60~90秒 |
| 进阶锦囊 | 推广内容层 | 解决具体痛点 | 技术论坛（ruyisdk.cn） | 标题格式：`【RuyiSDK进阶锦囊】<问题描述>` |

---

## 2. 各类型内容制作规范

### 2.1 官方文档修改

- **仓库**：`ruyisdk/docs`
- **流程**：Fork → 修改 → PR → 至少一位研发 approve → 合并
- **规范**：
  - 保持中文，技术术语可保留英文
  - 每个命令/配置必须有可验证的例子
  - 更新 `_sidebar.md` 或目录文件

### 2.2 工具使用案例

- **存放位置**：`ruyisdk/board-docs` 按板卡分类
- **模板**：

```markdown
# 案例：使用 ruyi 在 QEMU 上运行 CoreMark

## 场景描述
（一句话说明要做什么）

## 环境要求
- 主机 OS，已安装 ruyi >=x.x

## 步骤
1. `ruyi install ...`
2. `ruyi venv ...`
3. ...

## 预期输出
（截图或关键日志）

## 常见问题
- Q: ... A: ...
```

- **提交**：PR 到 `ruyisdk/board-docs`，需测试同学验证通过

### 2.3 实战体验包（含文档）

一个完整的体验包包括：
- **案例文档**（使用下方“实战五步法”模板）
- **可分发资源包**（注册到 `ruyisdk/packages-index`）

**案例文档模板（实战五步法）**：

```markdown
# 【RuyiSDK 实战】<场景> + <板卡> + <SoC> + <OS>

**作者**：@GitHubID
**验证环境**：ruyi x.x.x, 主机 OS
**预计耗时**：xx 分钟

## 1. 目标
（一句话说明）

## 2. 物料清单
- 硬件：板卡、SoC
- 软件：OS 镜像地址、工具链、源码地址

## 3. 环境配置
ruyi install ...
ruyi venv ...

## 4. 获取与构建
ruyi extract ...
make


## 5. 运行与验证
（预期输出截图）

## 6. 清理与体验包导出
ruyi package export ...


## 常见问题
```

**体验包五要素检查清单**（硬性要求）：

- [ ] 工具链/依赖声明（ruyi 版本、必要工具链）
- [ ] 系统镜像/版本指路（下载地址或来源）
- [ ] 示例工程代码库（可一键导入的仓库）
- [ ] 一键运行脚本/命令（可复制粘贴完成全流程）
- [ ] 已验证环境清单（板卡型号、SoC、OS 版本，已知限制）

**提交流程**：
1. 在个人环境跑通，记录全部命令和输出
2. 撰写案例文档（可先放个人仓库或 Notion）
3. 在 GitHub Issue 中贴出草稿链接，邀请至少两人试跑验证
4. 验证通过后，同时提交两个 PR：
   - PR1：案例文档 → `ruyisdk/board-docs`
   - PR2：体验包注册 → `ruyisdk/packages-index`
5. 合并后，将案例链接和体验包安装命令更新到官网“开发板应用案例”板块

### 2.4 入门指南系列（推广用）

- **形式**：图文帖（800字以内，5到8张截图）或短视频（60到90秒）
- **发布渠道**：
  - 首选：RuyiSDK 官方论坛（ruyisdk.cn）
  - 次选：知乎、掘金、CSDN、B站、抖音、微信公众号
- **内容选题**：《5分钟学会 ruyi 安装》《VS Code 插件跑通第一个 RISC-V 程序》
- **强制要求**：文末或视频简介必须附上“更多官方文档：ruyisdk.cn/docs”以及相关案例链接
- **标题格式**：`【{工具名} 入门指南】{具体内容}`

### 2.5 进阶锦囊

**核心原则**：论坛优先，快速响应，定期归档。

**步骤**：

1. **在技术论坛发帖**：https://ruyisdk.cn
2. **标题格式**：`【RuyiSDK进阶锦囊】<一句话问题描述>`
   - 示例：`【RuyiSDK进阶锦囊】ruyi venv 激活失败怎么办？`
3. **内容模板**：

```markdown
## 🔥 现象
（描述错误或异常行为）

## 🧠 原因
（简要分析）

## ✅ 解决方案
（可执行的命令或步骤）

## 💡 延伸建议（可选）
```

4. **添加标签**：`进阶锦囊`、工具标签（`ruyi`、`vscode-plugin` 等）
5. **互动确认**：至少一位用户回帖确认“按此方案解决了问题”后，该锦囊视为有效
6. **定期归档**：每月由维护者将已验证的锦囊帖子整理为 Markdown 文档，提交 PR 到 `ruyisdk/docs/faq/`，原帖末尾会追加“本问题已归档至官方 FAQ：[链接]”

## 3. 仓库与渠道总览

| 类型 | 仓库/渠道地址 | 主要内容 | Maintainer（角色） | 更新方式 |
|------|----------------|----------|---------------------|-----------|
| **文档仓库** | `ruyisdk/docs` | 官方安装指南、命令手册、配置说明、FAQ | 研发（技术审核）+ PM（规划） | PR 审核合并 |
| **案例仓库** | `ruyisdk/board-docs` | 工具使用案例、实战体验包文档 | 测试（验证）+ 实习生/社区（编写） | PR 审核合并 |
| **体验包注册仓库** | `ruyisdk/packages-index` | 体验包的安装定义（yaml） | 研发（审核）+ 案例作者 | PR 审核合并 |
| **管理仓库** | `ruyisdk/ruyisdk` | Meta 仓库，任务管理、Issue 追踪 | PM + 运营 | Issue + Project |
| **官网仓库** | `ruyisdk/ruyisdk-website` | 首页、文档聚合、案例展示、博客 | 运营 + PM + 前端志愿者 | PR 审核合并 |
| **推广渠道** | ruyisdk.cn, B站, 知乎, 掘金, CSDN, 公众号 | 图文帖、短视频、直播切片 | 运营（分发）+ PM（策略） | 直接发布 |

## 4. 如何认领任务与提交内容

### 4.1 任务认领（GitHub Issues）

- 在 `ruyisdk/ruyisdk` 仓库的 Issues 列表中，查找带 `help wanted` 或 `good first issue` 标签的任务
- 评论 `/assign` 认领，任务进入 `In Progress` 状态
- 如果没有现成 Issue，可以自己创建 Issue 描述你想写的内容，打上 `draft` 标签

### 4.2 草稿阶段（宽松协作）

- 在个人 GitHub 仓库、GitHub Gist、Notion 或任何可公开访问的地方撰写草稿
- 在对应的 Issue 下评论贴出草稿链接，邀请其他人评论
- 打上 `draft` 标签

### 4.3 正式提交（PR 阶段）

- 内容成熟后（至少一位非作者确认可用），按上表对应的仓库发起 PR
- PR 模板中需关联 Issue 编号，附上验证截图
- 技术内容需研发 approve；非技术内容（锦囊整理、推广记录）可由运营或任何已有提交记录的贡献者 approve
- PR 合并后，将 Issue 状态改为 `Done`

> 如果原作者不愿意自己提交 PR，可以授权其他人代为提交，PR 描述中注明 `Original author: @xxx`，并在文档头部保留 `authors` 字段。

## 5. 质量检查清单

提交前请确认以下项目：

- [ ] 所有命令或操作步骤均在指定版本上验证通过，并提供截图或关键输出
- [ ] 无错别字，中英文之间加空格（如：使用 `ruyi` 命令）
- [ ] 代码块标注了语言（如 `bash`、`c`）
- [ ] 引用了官方文档或外部链接时，URL 有效
- [ ] 文末包含反馈渠道（如：请到 GitHub Issues 提交问题）
- [ ] 元数据完整：标题、作者、验证版本、最后更新日期（Markdown 头部 YAML 格式）

## 6. 常见问题（FAQ）

**Q: 我没有开发板，能做什么？**
A: 可以写工具使用案例（在 QEMU 上验证）、进阶锦囊、入门指南图文/视频。

**Q: 我的英文不好，只能写中文可以吗？**
A: 可以，官网文档和社区内容目前以中文为主。

**Q: 视频录制需要专业设备吗？**
A: 不需要。手机 + 免费软件（OBS 录屏，剪映剪辑）即可。

**Q: 我写的内容被修改了，署名还在吗？**
A: 会在文件头部保留 `authors` 字段；大幅重写时会征求原作者同意。

**Q: 如何知道自己适合做什么类型的任务？**
A: 看 **规划书** 中的“不同角色如何参与”表格，结合自己的兴趣选择。也可以在 Issue 中留言，管理员会协助推荐。

**Q: 我遇到一个高频问题，但不想写长文怎么办？**
A: 直接发论坛帖打上 `【RuyiSDK进阶锦囊】` 标题，300-500 字即可，运营会定期整理归档到官方 FAQ。

## 7. 开始行动

1. 通读本指南、规划书和运营策略规范。
2. 打开 `ruyisdk/ruyisdk` 仓库的 Issues，找一个 `good first issue` 评论 `/assign`。
3. 按照对应类型的规范撰写草稿，发布后贴到 Issue 下。
4. 邀请至少一人试读/试跑，收集反馈。
5. 发起 PR，合并后你的贡献就成功汇入 RuyiSDK 生态。

**感谢你的参与，每一篇文档、每一个案例、每一条锦囊，都在帮助 RuyiSDK 变得更好用、更易用！**

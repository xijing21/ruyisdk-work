# RuyiSDK 内容建设者实操指南

本指南面向所有参与 RuyiSDK 内容建设的人员（研发、测试、运营、PM、实习生、社区贡献者）。

**请先阅读《RuyiSDK 内容建设规划书》**，了解整体架构、工作包和协作模式，再按本文档操作。

## 1. 内容类型与对应规范

根据规划书的三层架构，我们主要维护四类内容（后两类对应“案例/实战层”和“推广内容层”）：

| 内容类型 | 对应层次 | 用途 | 推荐长度 | 托管仓库 |
|----------|----------|------|----------|----------|
| 官方文档（修正/新增） | 官方文档层 | 权威参考 | 不限 | 文档仓库 |
| 工具使用案例 | 案例/实战层 | 教某个工具怎么用 | 500~1500字 | 案例仓库（文档部分） |
| 实战体验包（含文档） | 案例/实战层 | 完整场景复现 | 完整案例文档 + 可安装包 | 案例仓库 + packages-index |
| 入门指南（图文/视频） | 推广内容层 | 短平快引流 | 图文 800字以内；视频60~90秒 | 推广渠道（论坛、B站等） |
| 进阶锦囊卡 | 推广内容层 | 解决具体痛点 | 200~500字 | 社区锦囊仓库 |

### 1.1 官方文档修改

- **仓库**：`ruyisdk/ruyisdk-docs`（或 `ruyisdk/ruyisdk-website`）
- **流程**：Fork → 修改 → PR → 至少一位研发 approve → 合并
- **规范**：
  - 保持中文，技术术语可保留英文。
  - 每个命令/配置必须有可验证的例子。
  - 更新 `_sidebar.md` 或目录文件。

### 1.2 工具使用案例

- **存放位置**：`ruyisdk/ruyisdk-docs/cases/` 目录下
- **模板**：

```markdown
# 案例：使用 ruyi 在 QEMU 上运行 CoreMark

## 场景描述
（一句话说明要做什么）

## 环境要求
- 主机 OS，已安装 ruyi >=x.x

## 步骤
1. `ruyi install ...`
2. `ruyi venv ...`
3. ...

## 预期输出
（截图或关键日志）

## 常见问题
- Q: ... A: ...
```

- **提交**：PR 到文档仓库的 `cases/` 目录，需测试同学验证通过。

### 1.3 实战体验包（含文档）

一个完整的体验包包括：
- **案例文档**（使用下方“实战五步法”模板）
- **可分发资源包**（注册到 `ruyisdk/packages-index`，用户可通过 `ruyi install` 获取）

**案例文档模板（实战五步法）**：

```markdown
# 【RuyiSDK 实战】<场景> + <板卡> + <SoC> + <OS>

**作者**：@GitHubID  
**验证环境**：ruyi x.x.x, 主机 OS  
**预计耗时**：xx 分钟

## 1. 目标
（一句话说明）

## 2. 物料清单
- 硬件：板卡、SoC
- 软件：OS 镜像地址、工具链、源码地址

## 3. 环境配置
ruyi install ...
ruyi venv ...


## 4. 获取与构建
ruyi extract ...
make

## 5. 运行与验证
（预期输出截图）

## 6. 清理与体验包导出
ruyi package export ...

## 常见问题
```

**体验包注册**：
- 将生成的包上传至可访问的 URL，然后在 `ruyisdk/packages-index` 仓库中添加一个 `pacakges/<name>.yaml`，定义包名、版本、依赖、下载地址。
- 用户即可通过 `ruyi install <package-name>` 一键获取。

**提交流程**：
1. 在个人环境跑通，记录全部命令和输出。
2. 撰写案例文档（可先放个人仓库或 Notion）。
3. 在 GitHub Issue 中贴出草稿链接，邀请至少两人试跑验证。
4. 验证通过后，同时提交两个 PR：
   - PR1：案例文档 → `ruyisdk/ruyisdk-docs/cases/`
   - PR2：体验包注册 → `ruyisdk/packages-index`
5. 合并后，将案例链接和体验包安装命令更新到官网“开发板应用案例”板块。

### 1.4 入门指南系列（推广用）

- **形式**：图文帖（800字以内，5~8张截图）或短视频（60~90秒）。
- **发布渠道**：
  - 首选：RuyiSDK 官方论坛（forum.ruyisdk.cn）
  - 次选：知乎、掘金、CSDN、B站、抖音、微信公众号
- **内容选题**：例如《5分钟学会 ruyi 安装》《VS Code 插件跑通第一个 RISC-V 程序》。
- **强制要求**：文末或视频简介必须附上“更多官方文档：ruyisdk.cn/docs”以及相关案例链接。
- **提交流程**：不需要 PR 到官方仓库，但需在 `ruyisdk/community/promotion/` 目录下创建一个 `.md` 文件记录发布链接（便于汇总）。运营同学负责分发。

### 1.5 进阶锦囊卡

- **仓库**：`ruyisdk/community/tips/`
- **格式**（锦囊卡模板）：

```markdown
# [锦囊] 标题：一句话概括问题

**适用版本**：ruyi >=x.x  
**场景**：什么情况下会遇到

## 现象
（错误信息或异常行为）

## 原因
（简要解释）

## 解决方案
（步骤或命令）

## 标签
#ruyi #troubleshooting
```

- **来源**：GitHub Issues、社群聊天、双周报、自己踩坑。
- **提交流程**：直接 PR 到 `ruyisdk/community/tips`，至少一人 review（可以是运营或任意贡献者），合并后自动生成标签索引页。

## 2. 仓库与渠道总览（对应规划书各层次）

| 类型 | 仓库/渠道地址 | 主要内容 | Maintainer（角色） | 更新方式 |
|------|----------------|----------|---------------------|-----------|
| **文档仓库** | `ruyisdk/docs` | 官方安装指南、命令手册、配置说明 | 研发（技术审核）+ PM（规划） | PR 审核合并 |
| **案例仓库**（文档部分） | `ruyisdk/board-docs` | 工具使用案例、实战体验包文档 | 测试（验证）+ 实习生/社区（编写） | PR 审核合并 |
| **体验包注册仓库** | `ruyisdk/packages-index` | 体验包的安装定义（yaml） | 研发（审核）+ 案例作者 | PR 审核合并 |
| **锦囊卡仓库** | `ruyisdk/community/tips` | 进阶锦囊卡（Markdown） | 运营 + 社区贡献者 | PR 审核合并 |
| **推广素材记录仓库** | `ruyisdk/community/promotion` | 入门指南/视频的发布链接汇总 | 运营 | 直接 push 或 PR |
| **官网仓库** | `ruyisdk/ruyisdk-website` | 首页、文档聚合、案例展示、博客 | 运营 + PM + 前端志愿者 | PR 审核合并 |
| **推广渠道** | ruyisdk.cn, B站, 知乎, 掘金, CSDN, 公众号, 社群 | 图文帖、短视频、直播切片 | 运营（分发）+ PM（策略） | 直接发布，无需 PR |

> **注意**：所有仓库默认采用 Apache-2.0 许可证，文档类也可采用 CC BY-SA 4.0。贡献者需在 PR 中签署 DCO。

## 3. 如何认领任务与提交内容

### 3.1 任务认领（GitHub Issues）

- 在 `ruyisdk/community` 仓库的 Issues 列表中，查找带 `help wanted` 或 `good first issue` 标签的任务。
- 评论 `/assign` 认领。任务会进入 `In Progress` 状态。
- 如果没有现成 Issue，也可以自己创建 Issue 描述你想写的内容，打上 `draft` 标签。

### 3.2 草稿阶段（宽松协作）

- 在个人 GitHub 仓库、GitHub Gist、Notion 或任何可公开访问的地方撰写草稿。
- 在对应的 Issue 下评论贴出草稿链接，邀请其他人评论。
- 打上 `draft` 标签。

### 3.3 正式提交（PR 阶段）

- 内容成熟后（至少一位非作者确认可用），按上表对应的仓库发起 PR。
- PR 模板中需关联 Issue 编号，附上验证截图。
- 技术内容需研发 approve；非技术内容（锦囊卡、推广记录）可由运营或任何已有提交记录的贡献者 approve。
- PR 合并后，将 Issue 状态改为 `Done`。

> 如果原作者不愿意自己提交 PR，可以授权其他人代为提交，PR 描述中注明 `Original author: @xxx`，并在文档头部保留 `authors` 字段。

## 4. 质量检查清单（所有内容通用）

提交前请确认以下项目：

- [ ] 所有命令或操作步骤均在指定版本上验证通过，并提供截图或关键输出。
- [ ] 无错别字，中英文之间加空格（如：使用 `ruyi` 命令）。
- [ ] 代码块标注了语言（如 `bash`、`c`）。
- [ ] 引用了官方文档或外部链接时，URL 有效。
- [ ] 文末包含反馈渠道（如：请到 GitHub Issues 提交问题）。
- [ ] 元数据完整：标题、作者、验证版本、最后更新日期（Markdown 头部 YAML 格式）。

## 5. 常见问题（FAQ）

**Q: 我没有开发板，能做什么？**  
A: 可以写工具使用案例（在 QEMU 上验证）、进阶锦囊卡、入门指南图文/视频。

**Q: 我的英文不好，只能写中文可以吗？**  
A: 可以，官网文档和社区内容目前以中文为主。

**Q: 视频录制需要专业设备吗？**  
A: 不需要。手机 + 免费软件（OBS 录屏，剪映剪辑）即可。

**Q: 我写的内容被修改了，署名还在吗？**  
A: 会在文件头部保留 `authors` 字段；大幅重写时会征求原作者同意。

**Q: 如何知道自己适合做什么类型的任务？**  
A: 看 **规划书** 中的“不同角色如何参与”表格，结合自己的兴趣选择。你也可以在 Issue 中留言，管理员会协助推荐。

## 6. 开始行动

1. 通读本指南和《RuyiSDK 内容建设规划书》。
2. 打开 `ruyisdk/community` 仓库的 Issues，找一个 `good first issue` 评论 `/assign`。
3. 按照对应类型的规范撰写草稿，发布后贴到 Issue 下。
4. 邀请至少一人试读/试跑，收集反馈。
5. 发起 PR，合并后你的贡献就成功汇入 RuyiSDK 生态。

**感谢你的参与，每一篇文档、每一个案例、每一条锦囊，都在帮助 RuyiSDK 变得更好用、更易用！**

# RuyiSDK 内容建设规划书

## 一、背景与问题

RuyiSDK 已发布 CLI 工具、VS Code/Eclipse 插件及基础文档，但内容体系存在三方面短板：

1. **官方文档缺乏定期维护与逻辑重构**：新手查找信息路径长，部分章节过时。
2. **各工具的使用案例稀少**：用户不会用、遇到问题无处可查。
3. **缺少场景化实战体验包**：无法直观展示 RuyiSDK “从零到一”解决真实开发问题的能力。

> 内容建设是社区建设的重点，是产品推广的基石，不是某一个人的事，需要**研发、测试、运营、PM、实习生、社区贡献者**共同参与，大家一起贡献。

## 二、内容体系三层架构

| 层次 | 定位 | 受众 | 主要形式 | 主要维护者 |
|------|------|------|----------|------------|
| **官方文档层** | 权威、稳定、完整参考 | 所有用户 | 结构化文档、命令行手册 | 研发 + 文档志愿者 |
| **案例/实战层** | 可复现、场景化、高价值 | 开发者、评估者 | 完整案例文档 + 体验包（含代码/镜像） | 测试 + 实习生 + 社区 |
| **推广内容层** | 短平快、易传播、引流 | 潜在用户、社群 | 入门指南、锦囊卡、短视频 | 运营 + PM + 实习生 |

**三层关系**：
- 推广内容 → 吸引流量 → 引向官方文档和实战案例
- 实战案例 → 展示能力 → 可沉淀为官方文档
- 官方文档 → 持续更新 → 为推广内容提供准确素材

## 三、我们具体要做什么（工作包清单）

### P0 – 必须启动的基础工作

**官网文档仓库重构**
- 梳理现有文档，优化目录结构和章节逻辑
- 建立“快速上手”→“工具参考”→“使用案例”→“常见问题”的清晰路径
- 需要：研发给出技术准确性建议，PM 组织评审，实习生/运营执行结构调整

**实战体验包试水（1个标杆案例）**
- 选定最简单可复现的场景（如 CoreMark on LicheePi 4A）
- 产出：完整文档 + 可分发体验包（通过 ruyi 安装）+ 短视频演示
- 需要：有开发板的人（借用/远程）验证步骤；运营协助录制视频；实习生撰写文档

### P1 – 快速见效的内容补充

**各工具使用案例集**
- 为 ruyi、VS Code 插件、Eclipse 插件分别补充 3~5 个典型用法案例
- 每个案例包含：场景描述、命令/操作步骤、预期输出、常见问题
- 需要：研发提供场景思路，测试/实习生跑通并截图，PM 审核

**入门指南系列（推广用）**
- 制作“5分钟学会 ruyi 安装”、“VS Code 插件跑通第一个 RISC-V 程序”等短内容
- 形式：图文帖（论坛、知乎、掘金）+ 60秒短视频（B站、抖音）
- 需要：运营主导文案和分发，实习生录制/截图，全员转发

### P2 – 持续沉淀与社区共建

**进阶锦囊帖子（避坑指南）**
- 来源：从技术社区（ruyisdk.cn）、GitHub Issues、社群聊天中整理高频问题
- 发布：`【RuyiSDK进阶锦囊】`系列帖子发布在技术社区（ruyisdk.cn），保持统一的标题格式
- 长期维护：定期将【RuyiSDK进阶锦囊】系列文章汇集到官方文档仓库的“常见问题”板块

**更多实战体验包扩展**
- 新增场景：AI 推理、GUI 显示、驱动开发、基准测试等
- 组合不同板卡（LicheePi、Milk-V、SiFive、QEMU）和 OS（openEuler、Debian、RevyOS）
- 需要：有板卡的人提供验证环境，多人协作完成文档和打包

## 四、不同角色如何参与

| 角色 | 可贡献的内容类型 | 预计投入（每周） | 参与方式 |
|------|----------------|----------------|----------|
| **研发** | 审核技术准确性；提供命令示例；协助复杂环境调试 | 1~2小时 | 响应 PR 中的技术问题；标注文档中的过时描述 |
| **测试** | 跑通案例并截图；记录真实踩坑过程；验证体验包可复现性 | 2~3小时 | 在测试设备上执行并输出日志；提交锦囊素材 |
| **运营** | 撰写/润色推广文案；制作短视频；分发到外部平台 | 3~5小时 | 每周选1篇内容加工成图文/视频，发布并统计反馈 |
| **PM** | 规划优先级；协调资源；组织文档评审会 | 2~4小时 | 定期检查文档缺口；推动 P0 事项闭环 |
| **实习生** | 执行具体写作、截图、录制；认领锦囊和案例 | 全职/半职 | 按实操指南产出内容，发起 PR |
| **社区贡献者** | 提供新场景/板卡验证；编写锦囊；翻译 | 任意 | 通过认领 Issue 或直接 PR 提交 |

> 内容贡献者可以单独完成一个内容的全流程；也可以采用多人合作方式（例如：测试跑通 → 实习生写文档 → 运营发帖 → 研发确认技术点）。欢迎任何角色认领或协作。

## 五、协作模式与流程（简化版）

采用 **两阶段 + 透明看板**：

1. **草稿阶段**
   - 任何人在个人仓库/在线文档起草内容，在 GitHub Issue 中贴出链接，打上 `draft` 标签。
   - 其他人可以评论建议。无需正式 PR。

2. **正式归档阶段**
   - 内容成熟后（至少一位非作者确认可用），由任意成员发起 PR 到官方文档/案例仓库。
   - 技术内容需研发 approve，非技术内容需运营 approve。
   - PR 合并后更新官网。

3. **任务看板**
   - 使用 GitHub Projects 维护任务列表：`待认领 → 草稿中 → 待审核 → 已发布`
   - 每周同步一次进度（可在双周会或异步文档中更新）。

## 六、成功标志（方向性目标）

- 官网文档结构清晰，用户能3步内找到所需信息。
- 至少有 5 个工具使用案例和 2 个完整实战体验包可公开获取。
- 外部社区开始有人主动提交锦囊或案例。
- 推广内容在技术论坛/社群的月均阅读量 > 1000，短视频播放量 > 2000（暂定，可调）。

## 七、号召

内容建设不是额外工作，而是让 RuyiSDK 被更多人用起来的关键投资。**每个人都能贡献5分钟到5小时**：

- 研发：下次修 bug 时顺便更新一下对应的文档章节。
- 测试：跑完回归测试后，挑一个场景写成案例。
- 运营：每周把一条社群问答变成锦囊。
- PM：每月组织一次文档“清理日”。
- 实习生：把学习笔记整理成入门指南。
- 社区用户：分享你用 RuyiSDK 完成的小项目。

我们不需要完美的计划，只需要**每个人开始动笔、截图、录屏、发帖**。所有内容都存放在 GitHub，通过 PR 汇聚，最终沉淀为 RuyiSDK 的生态资产。

# RuyiSDK生态建设-内容建设篇
## 1. 为什么写这个？（目的与核心原则）
本文档旨在为RuyiSDK团队提供清晰的内容建设与推广指南。我们推行**“全员运营体系”**，内容不是某个运营的闭门造车，而是全员（特别是技术骨干）共同参与的价值输出。
- **核心原则：** 严格沿用并升级 proposal-001 规范，遵循 **“技术发起定调 + 宣发策划定级 + 按级协同创校 + 渠道矩阵分发 + 数据复盘反哺”** 的全员运营流程：
  - **技术发起定调**：成果达成时，技术负责人输出《成果价值自述》（痛点/动作/数据/影响），作为内容源头。
  - **宣发策划定级**：宣发人员结合 AI 辅助与行业视野，提出 S/A/B/C 分级建议与内容形态策划。
  - **按级协同创校**：按级别分工制作（A/S宣发主笔长文，B宣发出脚本技术跑Demo，C宣发直接提取），技术负责“事实与边界校对”，杜绝外行话与过度承诺。
  - **渠道矩阵分发**：严格按平台台账匹配渠道，一鱼多吃，形态适配。
  - **数据复盘反哺**：追踪阅读/下载/报错数据，评估影响力，反哺下一轮技术规划与选题。
- **统一品牌出口**：所有对外内容统一使用“RuyiSDK”，避免子品牌混淆。
- **目标导向**：让全体明白**为什么做内容**（提升影响力、降低支持成本），**怎么做内容**（按系列规范产出），**做完了怎么推**（多渠道组合分发）。

## 2. RuyiSDK生态建设-内容建设策略
我们的内容建设当前阶段聚焦两大核心策略，它们相辅相成，形成飞轮：
**策略一：夯实技术内容，筑基引流**
- **核心动作**：加强技术文档建设，优先支持热门且主流的开发板。
- **逻辑**：技术文档和开发案例是RuyiSDK的“地基”。没有能跑通的案例，推广就是空中楼阁。主流开发板自带流量，支持它们就是最高效的SEO和获客手段。
**策略二：抓企业/社区合作，破圈放大**
- **核心动作**：与开发板厂商、OS社区、上游工具链社区建立联合宣发。
- **逻辑**：基于策略一跑通的“开发板实战案例”，拿结果去跟企业谈合作。将单方面的“适配”升级为双边的“官方支持”，借助合作伙伴的渠道破圈，互相背书，成倍放大影响力。
> **飞轮效应**：技术案例跑通 -> 拿案例找企业谈合作 -> 联合宣发扩大受众 -> 吸引更多开发者产出更多案例 -> 滚雪球前行。

## 3. 内容及系列设计（含对应推广渠道简介）
将内容分为“技术实战”与“商业生态”两大主线。技术内容强调**高频细分**（小更新也能发），商业内容强调**声量标杆**（出新闻）。

### 3.1 主线一：技术内容建设（高频、实用、引流）
*设计思路：通过系列化命名，让每个小更新、小技巧都有归属，增加曝光频率，保持社区活跃度。*

| 系列名 | 定位与适用场景 | 产出形态 | 主要分发渠道 |
|-------|-------------|---------|-----------|
| 【{工具名} 小白书】/【{工具名} 入门指南】/【{工具名} 快速上手】 | 工具基础使用、新增功能/特性速览。<br>*(解决：ruyi包管理器发版如何高频曝光)* | 图文教程（含环境、步骤、避坑） | 官网文档、论坛教程区、公众号次条 |
| 【{工具名} 秘籍】/【{工具名} 进阶锦囊】/【{工具名} 避坑指南】 | 高阶技巧、快捷操作、特定报错解决。<br>*(解决：碎片的Tips也能成为流量入口)* | 短图文/卡片式 | 论坛、社群（微信群/QQ群）、公众号次条 |
| 【RuyiSDK 实战】 | 综合应用：板卡+SoC+OS+场景全链路打通。<br>*(这是核心弹药库)* | **三件套**：图文+视频+快速体验包(Registry) | 官网案例区、B站、论坛、Registry首页推荐 |
| 【Registry 新资源公示】 | packages-index新增资源汇总。<br>*(解决：小更新撑不起大稿，但需公示)* | 汇总列表（含一句话价值+安装命令） | 论坛公告、社群播报、双周进展 |
| 【RuyiSDK 兼容性一览】 | CI/CD与多平台适配成果资产化。<br>*(解决：基础设施建设如何体现外部价值)* | 结构化文档（已验证环境+已知问题） | 官网高级文档、公众号深度文章 |

### 3.2 主线二：商业合作/生态内容（标杆、背书、破圈）
*设计思路：基于合作，基本都能产出可发公众号头条的新闻，对互相提升影响力非常有好处。*

| 系列名 | 定位与适用场景 | 产出形态 | 主要分发渠道 |
|-------|-------------|---------|-----------|
| **【RuyiSDK 生态】** | 合作接入公告、联合通稿、活动报道。 | 官方新闻稿/海报 | 公众号头条、官网新闻、合作方媒体矩阵 |
| **【RuyiSDK 生态·联合案例】** | 标杆级商业落地全方案。<br>*(由【RuyiSDK实战】升级而来)* | 深度长文+客户背书+架构图 | 公众号头条、官网首页、行业大会演讲 |
| **【RuyiSDK 构建与分发】** | 强调GNU/LLVM/QEMU等上游二进制打包的价值。<br>*(替用户干脏活累活的价值输出)* | 技术博客/白皮书 | 官网、上游社区渠道、公众号 |
| **【RuyiSDK 故事】** | 个人开发者或企业用户的亲历者视角。<br>*(替代高门槛的“伙伴说”)* | 访谈图文/视频 | 公众号、论坛、B站 |

### 3.3 内容命名与标签规范（确保可检索、可聚类）

为避免各平台标题和标签混乱，导致搜索引擎/AI 无法聚类，统一执行以下规范：

- **实战类标题公式**【RuyiSDK 实战】{应用场景} + {板卡型号} + {SoC} + {OS}示例：【RuyiSDK 实战】AI推理 + LicheePi 4A + TH1520 + openEuler
- **入门/秘籍类标题公式**
  - 【{工具名} 入门指南】/【{工具名} 快速上手】：面向新用户
  - 【{工具名} 避坑指南】/【{工具名} 进阶锦囊】：面向有基础的用户
- **必打标签体系**任何跨平台发布的内容，必须包含以下 5 类标签（可多选）：
  1. `#RuyiSDK`
  2. `#板卡品牌型号`（如 `#LicheePi4A`、`#Milk-V-Duo`）
  3. `#SoC/架构`（如 `#TH1520`、`#RISC-V64`）
  4. `#操作系统`（如 `#openEuler`、`#Deepin`）
  5. `#应用场景`（如 `#AI推理`、`#嵌入式`）
     
  同一案例在官网、B站、论坛发布时，标签必须保持一致，便于跨平台聚合。

### 3.4 快速体验包交付标准（P0 实战案例的硬性门槛）

**定义**：快速体验包 = 让用户在 30 分钟内从“拿到板子”到“跑通第一个示例”的最小交付物。
**要求**：任何一个【RuyiSDK 实战】案例，必须配套上架 Registry 的体验包卡片，且必须包含以下五要素（缺一不可，否则宣发有权拒绝推广）：

1. **工具链/依赖声明**
   - 明确 ruyi 版本、必要工具链（如 gcc、qemu）、依赖库版本。
2. **系统镜像/版本指路**
   - 给出镜像下载地址或来源（如官方镜像链接、版本号）。
3. **示例工程代码库**
   - 提供可一键导入的示例工程仓库（含 README）。
4. **一键运行脚本/命令**
   - 必须有一段可复制粘贴的命令，完成从安装到运行的全流程。
5. **已验证环境清单（兼容性打标）**
   - 列出已验证的板卡型号、SoC、OS 版本，以及已知限制或坑点。
     体验包在 Registry 的卡片，必须包含：一键命令、已验证环境、反馈/提问入口（跳转论坛或 Issue 模板）。

### 3.5 社区共创与精选机制（给 UGC 留出口）

当前以 PGC 为主，但长期必须让社区帮我们写，否则难以规模化。

- **【RuyiSDK 社区精选】** 作为**叠加标签**（不单独开专栏）
  - 任何用户在论坛写的好文，只要质量达标，运营打上此标签。
  - 公众号/B站做月度汇总转发，并在文章中注明“社区贡献”。
- **激励与反哺**
  - 被精选的用户，给予社区曝光及周边奖励。
  - 若是高质量实战案例，官方协助完善“快速体验包”，并升级为【RuyiSDK 生态·联合案例】候选。

## 4. 内容推广策略（含流程与经典案例）
推广不是内容写完后的“后续动作”，而是从产品发版前就应介入的“前置规划”。
### 4.1 推广前置流程（发版即宣发）
1. **发版规划期**：开发团队规划新版本/新支持板卡时，@运营入群。运营评估定级（S/A/B/C），提前策划宣发节奏。
2. **联调测试期**：运营跟进进度，撰写宣发文案初稿（提炼用户价值，而非堆砌技术参数）。
3. **发稿对齐**：开发审核文案技术事实，运营确认品牌规范，确定多渠道分发计划。
4. **发版宣发期**：代码提交/版本发布的同时，各渠道内容同步上线。
5. **数据复盘**：发布后1-2周，查看阅读/下载/报错数据，沉淀为下一轮优化方向。

### 4.2 一鱼多吃：一个话题的多渠道推广策略
同一个技术话题，在不同渠道的受众和阅读场景不同，**内容组织与形态必须适配**，切忌在所有平台发一模一样的长文。
**【经典案例说明】：以“RuyiSDK新增支持Milk-V Duo开发板”为例**
- **核心物料**：开发者成功在Duo上跑通RuyiSDK的实操过程及一键运行脚本。
- **渠道适配拆解**：
  1. **官网/文档（权威沉淀）**：
     - 形态：结构化图文教程+快速体验包入口。
     - 侧重：步骤严谨、命令准确，供用户随时查阅。
  2. **B站（视觉吸引）**：
     - 形态：3分钟快节奏实操视频。
     - 侧重：“开箱-敲命令-出结果”的视觉冲击，BGM轻快，字幕突出，少讲原理多展示结果。
  3. **公众号（价值解读）**：
     - 形态：深度推文。
     - 侧重：标题抓人（如《最低几十元玩转RISC-V！RuyiSDK一键点亮Milk-V Duo》），讲清楚Duo是什么，RuyiSDK解决了什么痛点（省去交叉编译烦恼），文末引导去Registry下载。
  4. **微信群/QQ群（即时触达）**：
     - 形态：一句话播报+海报/短链接。
     - 侧重：“RuyiSDK已支持Milk-V Duo，一键安装命令：xxx，体验包链接：xxx”，直截了当，方便复制。
    
### 4.3 内容发布基线节奏（避免三天打鱼两天晒网）

SOP 解决了单篇流程，但全局节奏也必须有底线。建议执行以下基线（可根据团队资源微调）：

- **C 级（日常维系）**：双周必有
  - 依托：双周进展、Registry 新资源公示、社群播报。
- **B 级（核心流量）**：每月至少 2 篇
  - 如：【ruyi 入门指南】、【ruyi 避坑指南】、【实战】普通案例。
- **A/S 级（势能爆发）**：每季度至少 1 次
  - 如：【RuyiSDK 生态·联合案例】、生态报告、大版本发布。
    低于此基线，视为内容建设亮黄灯，需要在下个周期补回。

### 4.4 联合宣发协作规范（Co-Branding）

与合作伙伴联合宣发时，若流程不清，极易被拖节奏或版本错位。统一执行以下红线：

1. **同步定调**
   - 与技术定调同步，明确双方审核人及承诺的审核 SLA（建议 24 小时内反馈）。
2. **首发对齐**
   - 必须约定精确到小时的发布时间，己方官网/公众号与对方渠道同步上线，避免“先发后不发”。
3. **互链承诺**
   - 己方文章必须带对方官网/购买链接；对方文章必须带 RuyiSDK Registry 链接，否则不构成“联合”。
4. **素材与口径统一**
   - 联合海报/通稿中的产品名、Logo、版本号，必须以双方确认的最终版本为准，不得单方面修改。

## 6. 闭环机制
内容建设不仅是“写和发”，还需要关注**资产沉淀与数据反哺**，这是很多人容易忽略的。

### 6.1 资产沉淀（内容不烂尾）

- 所有公众号发的教程、社群分享的技巧，最终都要**长链路沉淀到官网文档或Registry**。
- 公众号会沉底，但官网搜索永远在。不要让用户只能在微信群翻聊天记录找答案。
- **硬性要求**：任何对外发布的内容，文末必须有“永久地址”（官网/Registry/论坛贴），且该地址在可预见未来不会失效。

### 6.2 数据反哺（用数据指导下一步）

通过预设埋点和统计的指标，利用**易获取、成本低**的数据做反哺，并不断完善收集收集，提供更有指导意义的观察数据。

1. **Registry 侧可观察的数据**
  > 从Registry下载量和报错聚类，看哪个板卡最受欢迎、哪个环境最容易出错 -> 决定下一个【实战】写哪块板卡，下一个【兼容性一览】写哪个工具。
   
   - 各包/体验包的下载量（大致趋势即可）。
   - 高频报错/反馈在论坛/Issue 中的聚类。
   - 行动：
     - 某板卡体验包下载明显高 → 优先做更多【实战】和【联合案例】。
     - 某环境报错集中 → 产出【避坑指南】或【兼容性一览】。
       
3. **社群与论坛的“软数据”**
> 从阅读量，看用户喜欢看小白书还是秘籍 -> 调整内容配比。

   - 微信群/QQ群、论坛中高频出现的问题、板卡型号、OS 类型。
   - 行动：
     - 某开发板反复被问 → 优先补充该板的小白书/实战案例。
     - 某版本工具链反复出问题 → 推动技术侧修复，并产出对应秘籍/避坑指南。
     
4. **官网/示例站访问统计（后续建设）**
   - 当前 webanalytics 方案已规划使用 Umami + Cloudflare Analytics，以隐私友好的方式采集 PV/UV、开发板访问占比、来源渠道等。
   - 在此体系完全落地前，可先以“论坛+社群反馈”为主；待访问统计体系成熟后，再引入“阅读量/热门页面”等指标做更精细的选题决策。

### 6.3 开发板支持优先级矩阵

资源有限，技术内容建设必须排优先级。按照“生态热度+技术成熟度”评估，当前优先死磕 LicheePi 4A、Milk-V Duo 等少数标杆，打透再铺开（详见文档：*开发板支持优先级矩阵*，链接待补充）。

## 附录：新成员内容行动指南
如果你是新加入团队的开发或运营同学，请按照以下指引开始你的内容产出：
1. **在哪写？用什么工具？**
   - 技术文档/教程：统一在指定的代码仓库（如GitHub docs目录）使用Markdown编写，便于版本管理。
   - 公众号/长图文：运营在微信公众号后台或第三方排版工具中制作。
2. **有没有现成的模板？**
   - 有！写【小白书/入门指南】请使用 `[TPL-入门指南.md]` 模板；写【实战】请使用 `[TPL-实战案例.md]` 模板。模板规定了必须包含的章节（如环境准备、核心步骤、常见报错等），不要自由发挥遗漏关键信息。
3. **截图和命令有什么要求？**
   - 代码/命令：必须可直接复制运行，不要截图贴代码！不要省略前提路径！
   - 截图：关键结果必须有截图证明，截图需清晰，敏感信息需打码。
4. **写完找谁审？如何提交？**
   - 提交PR/MR到文档仓库。
   - 在RuyiSDK内容协同群中@对应模块的技术负责人进行技术审核，@运营同学进行润色和定级对齐。
5. **我想报错/补充小技巧，但不想写长文怎么办？**
   - 把现象、解决思路和命令发给运营同学，或直接在论坛发帖打上【{工具名} 秘籍】标签，运营会定期收录并整合成官方专栏。
6. **我参与的案例要上快速体验包，该怎么做？**
   - 确认你已经按照 3.3 节的“五要素”准备齐全：工具链/镜像/示例/一键脚本/已验证环境。
   - 在 PR 中附上 Registry 卡片截图或链接，运营会检查是否符合标准，不符合则退回补充。

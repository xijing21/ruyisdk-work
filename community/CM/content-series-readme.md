# RuyiSDK 内容系列贡献者指南
本指南面向所有愿意参与 RuyiSDK 内容建设的贡献者，目标是让三大内容系列：
- 【{工具名} 入门指南】
- 【{工具名} 进阶锦囊】
- 【RuyiSDK 实战】（快速体验包）
可以快速、协同地“填空”式完成，并逐步沉淀为一本可读的“RuyiSDK 电子书”。

---

## 1. 系列说明与定位
### 1.1 三大内容系列的角色
| 系列 | 核心读者 | 主要目标 | 与官方文档的关系 |
|------|----------|----------|------------------|
| 【{工具名} 入门指南】 | 零基础新手 / 刚接触该工具的 RuyiSDK 用户 | 让用户“从零到跑通第一个成功示例” | 补充官方“安装 + 命令/功能概览”之外的手把手实践和避坑，不是重复官方文档 |
| 【{工具名} 进阶锦囊】 | 已经能用该工具完成基础操作，想更稳、更快、更少踩坑的开发者 | 系统整理使用技巧、兼容性、版本差异和排查思路 | 把散落在 issue / 论坛 / 双周报里的经验结构化，为官方文档提供“实战输入” |
| 【RuyiSDK 实战】（快速体验包） | 有具体开发板/场景，想“直接照着做一遍”的开发者 | 提供“场景 + 板卡 + SoC + OS + 工具链 + 代码”的一键式可复现案例 | 把官方案例（如 LicheePi 4A coremark 案例）模板化，扩展更多场景和板卡 |

### 1.2 工具映射与命名
- 工具范围：
  - RuyiSDK 包管理器（CLI 命令 `ruyi`）
  - RuyiSDK VS Code 插件
  - RuyiSDK Eclipse 插件
- 命名规则：
  - 入门 / 进阶系列：  
    `【ruyi 入门指南】` / `【RuyiSDK VS Code 插件 入门指南】` / `【RuyiSDK Eclipse 插件 入门指南】`  
    标题中写“工具名”，方便检索和区分。
  - 实战系列：  
    标题中不再写工具名，而是用“场景 + 板卡 + SoC + OS”标识，如：  
    `【RuyiSDK 实战】AI推理 + LicheePi 4A + TH1520 + openEuler`  
    在文中用标签标明主要工具（例如：`RuyiSDK 包管理器`、`RuyiSDK VS Code 插件`、`RuyiSDK Eclipse 插件`）。

## 3. 每个小节的主要内容（给贡献者的“填空提示”）
为了让大家写起来更有方向，每篇文档的小节可以统一包含以下几类内容（具体可按章节取舍）：
- **目标**：这一节要解决什么问题？读者读完能做到什么？
- **前提条件**：需要先完成哪些章节 / 安装哪些软件？
- **步骤与命令**：
  - 对于 CLI：完整的命令行示例，包括参数说明；
  - 对于 GUI：界面路径 + 截图（如有条件）+ 关键选项说明。
- **输出与验证**：如何判断这一步成功了？典型输出示例。
- **常见问题**：列出 2–5 个高频错误，每个给出“现象 + 原因 + 解决”。
- **延伸阅读**：指向官方文档、相关进阶锦囊或实战案例。
在仓库中，可以为每个系列维护一个 `README.md`，里面只放大纲 + 每节“填空提示”，不写具体内容，避免频繁冲突。

- 每个系列一个目录，`README.md` 放目录和写作提示。
- 文件名使用小写 + 短横线，避免中文文件名。
- 每篇文档开头统一加元信息：
```markdown
---
title: 【ruyi 入门指南】3.2 ruyi install 安装工具链
series: ruyi-getting-started
order: 0302
authors: <your-GH-username>
reviewers: 
status: draft   # draft / reviewing / published
url: https://your-repo/.../0302-install-toolchain.md
---
```
### 4.2 目录及时更新 & 文档挂链接
- 每个系列的 `README.md` 中维护一个目录表，格式如：
```markdown
# 【ruyi 入门指南】目录
| 章节 | 文件 | 状态 | URL |
|------|------|------|-----|
```

- 约定：
  - 任何人新增 / 删除 / 合并章节时，**必须同步更新这个目录表**；
  - `URL` 字段允许：
    - 个人仓库 / GitHub Pages / 任意可访问地址；
    - 也允许 ruyisdk.cn 上的正式 URL。
  - 目标：只要按目录走，就能像“电子书”一样依次阅读，无论内容托管在哪。
### 4.3 内容生产与正式提交的折中方案
你担心的问题很实际：**流程太规范 → 时效性差 / 参与度低；太分散 → 内容容易不了了之**。  
建议采用“两阶段”策略：
1. **草稿阶段：宽松托管 + 社区评审**
   - 贡献者可以在：
     - 自己的 GitHub 仓库；
     - Gitee / GitCode 等国内平台；
     - 甚至 Notion / 飞书文档（只要可公开访问）。
   - 只需在对应系列的目录表中填上 `URL`，并给状态标为 `draft`。
   - 社区通过：
     - GitHub Issue / PR 评论；
     - ruyisdk.cn 论坛帖子；
     - 微信 / QQ 群（官方社群）  
     对草稿提出意见。
2. **正式阶段：逐步合入官方仓库**
   - 当某篇文档：
     - 经过至少一位非作者评审；
     - 内容与官方文档不冲突（或已计划替换）；
     - 许可证兼容（Apache-2.0 / CC-BY-SA 等）；
   - 则由作者或 RuyiSDK 成员发起 PR，提交到正式仓库（如 `ruyisdk/ruyisdk-website` 或新建 `ruyisdk/docs` 仓库）。
   - 正式仓库中的文档会被同步到 ruyisdk.cn 官方文档站。
### 4.4 内容生产者 ≠ 提交者时的著作权与署名
你问：**“内容生产者和提交者不一致，开源社区如何处理？”**  
RuyiSDK 项目已经采用 DCO（Developer Certificate of Origin）来规范贡献权属。常见做法：
1. **DCO + Signed-off-by**
   - 每个提交者需要在提交信息中加 `Signed-off-by:`，声明自己有权提交该贡献。
   - 对于文档，可以在 PR 模板中要求：
     - 作者签署 DCO；
     - 若提交者非作者，提交者需确认作者已同意采用 Apache-2.0 / CC-BY-SA 等开源许可证。
2. **作者与提交者都署名**
   - 在文档元信息中同时保留：
     - `authors:`：实际写内容的人；
     - `committers:` 或 `submitter:`：负责整理并提交 PR 的人。
   - 在文末或顶部加一行类似：
     ```markdown
     本文由 <author> 创作，由 <committer> 整理提交至 RuyiSDK 仓库，采用 Apache-2.0 许可证发布。
     ```
3. **许可证选择**
   - 建议所有内容统一采用与 RuyiSDK 代码仓库相同的许可证（Apache-2.0）。
   - 若作者希望采用 CC-BY-SA 等，需要在 PR 中明确说明，并经维护团队评估兼容性。
4. **不愿继续维护的内容**
   - 如果作者授权但不愿自己提交，可在 PR 中注明：
     - 作者仅授权，不再负责后续维护；
     - 由 RuyiSDK 成员或其它贡献者接手维护，但保留原作者署名。

## 5. 协同落地还需要补充的机制
除了你已经想到的目录更新 + URL 挂接，为了让“大家一起写”真正跑起来，建议再补几块：
### 5.1 Issue / PR 模板
- 在内容仓库创建：
  - `ISSUE_TEMPLATE/new-content.md`：申请新建某个章节 / 实战案例；
  - `ISSUE_TEMPLATE/review-content.md`：请求社区评审某篇草稿；
  - `PULL_TEMPLATE/zh.md`：文档提交模板，包含：
    - 关联的 Issue；
    - 是否涉及 DCO / 许可证变更；
    - 作者与提交者信息。
### 5.2 认领与进度追踪
- 在 `README.md` 或一个专门的 `PROGRESS.md` 中，用表格记录：
```markdown
| 系列 | 章节 | 认领人 | 状态 | 草稿 URL | 评审人 |
|------|------|--------|------|----------|--------|
| ruyi-getting-started | 03-install-ruyi | @someone | draft | https://... | @reviewer1 |
| ... | ... | ... | ... | ... | ... |
```
- 避免“多人同时写同一章节”的冲突，也方便后来者接手。
### 5.3 评审与合并标准
建议明确：一篇文档从 `draft` → `published` 至少满足：
1. 技术上：
   - 命令 / 界面路径在最新稳定版本上验证通过；
   - 与官方文档（如 ruyi README、VS Code 插件文档）不冲突。
2. 内容上：
   - 有明确的“目标 + 步骤 + 验证 + 常见问题”；
   - 无明显错漏，关键步骤有输出示例。
3. 规范上：
   - 符合仓库结构、命名和元信息要求；
   - 作者/提交者签署 DCO 或明确授权。
### 5.4 激励与反馈
- 可以在 ruyisdk.cn 社区定期：
  - 公布“新增内容 / 贡献者榜单”；
  - 对高质量实战案例作者，给予社区周边 / 荣誉头衔等。
- 每个实战案例最后附上：
  - 作者联系方式（如愿意）；
  - 欢迎反馈的渠道（Issue / 论坛链接）。

## 6. 一个“试水”案例的落地步骤（针对【RuyiSDK 实战】）
你提到想先做一个试水案例，把整个定义走通。建议：
1. 选定第一个实战：
   - 场景：基准测试（如 coremark，已有官方案例）；
   - 板卡：LicheePi 4A + TH1520；
   - OS：openEuler 或 RevyOS。
2. 按上面【2.6】的大纲写一篇，严格遵循：
   - 每步可复制粘贴；
   - 包含完整命令与输出；
   - 在文末给出“10 分钟复现”时间估计。
3. 放在个人仓库 / GitHub Pages，在 `ruyi-practice/README.md` 目录表中登记：
   - 标题：【RuyiSDK 实战】CoreMark + LicheePi 4A + TH1520 + openEuler；
   - 状态：`draft`；
   - URL：你的地址。
4. 在 RuyiSDK 社区（论坛 / Issue）发帖，邀请大家试跑并评论。
5. 根据反馈修一版，再提交到正式文档仓库，作为“样板案例”，后续其它实战都照此结构写。

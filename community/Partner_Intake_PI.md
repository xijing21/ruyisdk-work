### 合作建议与说明

1.  **深化现有伙伴合作**：
    *   针对“RuyiSDK 现有合作伙伴” 清单中的企业继续保持联系和合作关系。
    *   **Milk-V、Sipeed**：企业合作意向以及产品集成度都相对高，建议继续保持深度合作。且这两个企业是针对RISC-V架构的设备厂商，产品集中在 RISC-V 架构，建议长期关注和合作。
    *   芯片+设备均覆盖的研发企业，如玄铁、进迭时空等，也有良好的关系基础，可以在更广泛范围合作，如编译器等。
    
2. 针对2026新品发布计划，找准合作契机
    *   **Milk-V、Sipeed、Banana Pi、Orange Pi**：这几家在2026年均有明确的新品发布计划。建议RuyiSDK团队在其新品上市前，即开始进行工具链适配、系统镜像构建和软件包集成工作，确保RuyiSDK能第一时间全面支持这些新硬件。
    *   特别是**Milk-V Jupiter2 (K3芯片)** 和**DeepComputing DC-ROMA Mainboard III (K3芯片)**，作为首批RVA23合规芯片的产品，是展示RuyiSDK对最新RISC-V国际标准支持能力的绝佳平台。
    *   **进迭时空 (SpacemiT)**：其K3芯片将成为2026年多家厂商高端开发板的核心。建议与进迭时空建立更紧密的技术沟通，提前获取芯片详规，使其Bianbu OS与RuyiSDK生态更好地协同，甚至考虑将RuyiSDK工具链作为其官方推荐开发工具之一。
    
2.  **重点接洽新伙伴**：
    *   **南京沁恒微电子 (WCH)**：虽然其工具链自成一体（MounRiver Studio），合作难度较高 ，但其在MCU开发者中拥有巨大影响力。2026年新品**CH32H417**是高性能USB3.0 MCU，合作切入点可尝试：
        *   **方案1（轻量级）**：将RuyiSDK包管理器作为获取CH32V系列SDK、数据手册和例程的**可选渠道**，方便习惯使用命令行和开源工具的开发者。
        *   **方案2（深度合作）**：联合推出针对特定应用（如工业控制、物联网网关）的**参考设计**，其中RuyiSDK负责Linux端应用开发环境，WCH提供MCU端实时控制固件，展示异构开发的完整流程。
    *   **华创微 (HC Semi)**：作为国家队新秀，其高性能处理器和AI芯片代表了RISC-V在边缘计算的重要突破。鉴于其芯片刚流片成功，开发板尚未公布 ，这是**提前介入、建立标准**的最佳时机。建议主动接触，提供RuyiSDK作为其官方SDK的基础，或协助其构建初版BSP和开发文档，抢占生态高地。

3.  **关注操作系统厂商**：
    *   openEuler（for RISC-V）、openRuyi 是软件所维护的 RISC-V 操作系统，考虑一起联合与RISC-V 设备厂商展开合作。
    *   与 RuyiSDK 合作的设备厂商中自己维护自定义的操作系统的，可以进一步就 RuyiSDK 包管理器、编译工具链等软件展开合作，纳入定制系统的系统软件源，甚至预装软件中。
    *   对于**deepin、openKylin、OpenCloudOS**等已入驻的OS厂商，2026年的合作重点应从“入驻”转向“产品化”。确保RuyiSDK能够方便地安装这些操作系统，并针对新发布的开发板提供优化的系统镜像。

### 审核完善后的【合作建议与说明】

为在2026年高效推进RuyiSDK生态建设，我们建议从以下三个维度分层分类地开展合作伙伴工作：

#### 一、 存量合作深化：巩固现有伙伴关系，跟进新品集成

针对已建立合作关系的伙伴，工作重点应从“建立联系”转向“产品化与深度绑定”。

1.  **核心设备厂商（Milk-V / Sipeed）**：
    *   **现状**：双方合作基础深厚，产品集成度高（如RevyOS深度适配），是RISC-V架构的核心布道者。
    *   **2026年策略**：
        *   **无缝跟进新品**：针对 **Milk-V Titan (已开售)** 和 **Jupiter2系列 (已公布)**，在新品上市前启动适配工作，确保RuyiSDK能够第一时间提供完整的工具链支持和优化后的系统镜像，保持“首发支持”的优势。
        *   **联合市场活动**：围绕高性能计算（Titan的PCIe扩展能力）和RVA23新特性（Jupiter2的K3芯片），联合策划技术直播、博客或线下工作坊，共同向开发者社区传递价值。

2.  **芯片与工具链伙伴（玄铁 / 进迭时空 / 赛昉科技 / 嘉楠科技 / 算能科技）**：
    *   **现状**：玄铁在工具链层面有长期联合发布（如RV64ILP32）；进迭时空的K1/K3成为多家板商核心；赛昉、嘉楠、算能也已入驻社区。
    *   **2026年策略**：
        *   **进迭时空 (SpacemiT)**：其 **K3芯片** 是2026年RISC-V高端市场的明星产品。建议建立“技术直通车”，提前获取芯片详规，推动RuyiSDK工具链成为其Bianbu OS生态的官方推荐/预置开发环境。
        *   **玄铁 (T-Head)**：继续深化在开源工具链（如LLVM、GCC）和内核（如新32位、RV64ILP32）上的联合研发，将RuyiSDK作为玄铁生态新特性的首发验证平台。
        *   **赛昉科技 (StarFive) / 嘉楠科技 (Canaan) / 算能科技 (SOPHGO)**：针对其已集成和待集成的产品（如VisionFive 2 Lite、JH-7110、K510、SG2044等），建立定期技术同步机制，确保RuyiSDK包管理器中的BSP、内核和镜像始终保持最新，并适配其量产规划。

#### 二、 增量合作拓展：瞄准2026新品，抢占生态高地

针对已有明确新品计划但合作尚浅或尚未合作的伙伴，应主动出击，找准切入点。

1.  **紧跟新品发布计划（Banana Pi / Orange Pi / DeepComputing）**：
    *   **机会点**：**香蕉派 (BPI-CM6, BPI-RV2)**、**香橙派 (Orange Pi RV2, R2S)** 和 **DeepComputing (DC-ROMA Mainboard III)** 在2026年均有明确新品。这些厂商亟需稳定、易用的软件开发环境来提升其产品的开发者体验。
    *   **行动建议**：在FOSDEM 2026等展会后，立即与Banana Pi和Orange Pi建立联系，获取其新平台（Siflower SF21H8898, Ky X1）的技术资料，启动早期适配。对于DeepComputing的K3主板，可参考与进迭时空的合作模式，提供从固件到应用层的全栈支持。

2.  **突破潜在重量级伙伴（华创微 HC Semi）**：
    *   **机会点**：作为国家队新秀，其高性能处理器和AI芯片刚流片成功，正处于生态构建的**黄金窗口期**。此时介入，有机会帮助定义其官方SDK的标准。
    *   **行动建议**：由PLCT实验室或软件所高层牵头，主动接洽华创微。提议协助其构建基于RuyiSDK的参考BSP、开发文档和示例代码，将其芯片纳入RuyiSDK的官方支持列表，实现“芯片流片即支持”的生态效果。

3.  **探索生态互补型伙伴（南京沁恒微电子 WCH）**：
    *   **现状**：WCH工具链自闭环，在MCU领域拥有海量开发者，但合作切入点需要创新。
    *   **行动建议**：建议采用 **“轻量级切入，重场景示范”** 的策略。
        *   **轻量级**：将RuyiSDK包管理器打造为获取WCH官方SDK、数据手册和例程的**便捷渠道**，降低Linux/macOS开源开发者的入门门槛。
        *   **重场景**：针对其2026年新品 **CH32H417 (高性能USB3.0 MCU)**，联合推出“RISC-V异构计算参考设计”。例如：RuyiSDK负责在Linux开发板上（如Milk-V Jupiter）运行复杂应用，通过USB3.0与CH32H417进行高速数据交互，展示完整的解决方案。

#### 三、 生态协同共建：联合操作系统与软件厂商，构建完整闭环

1.  **操作系统厂商（deepin / openKylin / OpenCloudOS / 苦芽科技 / 秦派软件 / 英麒智能）**：
    *   **2026年策略**：从“入驻仪式”转向“产品化落地”。
        *   **镜像预置**：与Milk-V、Sipeed等硬件伙伴合作，在2026年新开发板的官方推荐操作系统列表中，优先预置或引导用户使用经过RuyiSDK优化后的deepin、openKylin等系统镜像。
        *   **工具链集成**：推动各OS厂商在其RISC-V版本的软件源中，默认集成RuyiSDK包管理器，并将其作为系统自带的开发环境配置工具，实现“开箱即用”。
        *   **联合运维**：与OS厂商建立联合CI/CD流水线，确保RuyiSDK的更新不会破坏其在主流开发板上的系统兼容性。

2.  **自有操作系统伙伴（Bianbu OS / openEuler RISC-V / openRuyi）**：
    *   **策略**：这些系统与RuyiSDK同源或关系紧密，是天然的深度合作伙伴。
    *   **行动建议**：将RuyiSDK包管理器直接纳入这些系统的**默认软件源**，甚至**预装**在系统镜像中。当开发者使用这些系统时，可以立即通过`ruyi`命令安装最新的工具链、仿真器或刷写系统镜像，形成“硬件（板厂）- 操作系统（OS厂）- 开发工具（RuyiSDK）”的生态闭环。

### 调整后的【合作建议与说明】（RevyOS维护调整变量修订版）

#### 一、存量合作深化：巩固现有伙伴关系，跟进新品集成

针对已建立合作关系的伙伴，工作重点应从“建立联系”转向“产品化与深度绑定”。

1.  **核心设备厂商（Milk-V / Sipeed）**：
    *   **现状**：双方合作基础深厚，过往RevyOS的深度适配是其核心软件生态之一。
    *   **2026年策略**：
        *   **做好过渡期沟通**：主动向Milk-V、Sipeed等伙伴说明RevyOS维护调整的背景，明确openRuyi将接棒继续提供高质量系统镜像支持，消除伙伴顾虑。
        *   **跨组协同机制**：建立RuyiSDK工具集团队与openRuyi团队的定期沟通机制。在跟进**Milk-V Titan、Jupiter2**等新品时，工具集适配与系统镜像构建需同步规划、协同发布，确保开发者拿到的是“工具+系统”的一体化体验。
        *   **联合市场活动**：围绕新品发布，策划工具集团队+openRuyi团队联合出镜的技术直播或博客，向社区传递“分工协作、合力支持”的清晰信号。

2.  **芯片与工具链伙伴（玄铁 / 进迭时空 / 赛昉科技 / 嘉楠科技 / 算能科技）**：
    *   **现状**：玄铁在工具链层面有长期联合发布；进迭时空的K1/K3成为多家板商核心；赛昉、嘉楠、算能已入驻社区。
    *   **2026年策略**：
        *   **进迭时空 (SpacemiT)**：其**K3芯片**是2026年RISC-V高端市场的明星产品。建议由RuyiSDK工具集团队对接芯片底层工具链优化，同时引入openRuyi团队对接其Bianbu OS生态，探讨openRuyi与Bianbu OS的技术协同或镜像互补。
        *   **玄铁 (T-Head)**：继续深化在开源工具链（如LLVM、GCC）和内核（如新32位、RV64ILP32）上的联合研发。同时，推动openRuyi成为玄铁生态的参考操作系统之一，形成“工具链+系统”的双重绑定。
        *   **赛昉科技 / 嘉楠科技 / 算能科技**：针对其产品线，RuyiSDK工具集负责BSP、内核的持续更新，openRuyi团队负责提供基于这些BSP的优化系统镜像。建议建立三方（芯片/板厂 + 工具集团队 + openRuyi团队）技术同步会机制。

#### 二、增量合作拓展：瞄准2026新品，抢占生态高地

针对已有明确新品计划但合作尚浅或尚未合作的伙伴，应主动出击，找准切入点。**此处的关键在于，向新伙伴清晰传达RuyiSDK工具集与openRuyi的分工定位，避免混淆。**

1.  **紧跟新品发布计划（Banana Pi / Orange Pi / DeepComputing）**：
    *   **机会点**：这些厂商亟需稳定、易用的软件开发环境和操作系统来提升其产品的开发者体验。
    *   **行动建议**：
        *   **分工定位沟通**：在初次接洽时，即向伙伴说明：RuyiSDK工具集负责提供包管理器、编译工具链、镜像刷写工具等开发环境；openRuyi（如适用）可提供开箱即用的系统镜像。两者可独立或组合使用。
        *   **技术对接**：由RuyiSDK工具集团队获取新平台（Siflower SF21H8898, Ky X1, K3）技术资料，完成基础工具链适配；同时将适配成果同步给openRuyi团队，由openRuyi团队评估是否需要构建官方系统镜像。
        *   **灵活合作模式**：允许伙伴根据自身需求，选择仅集成RuyiSDK工具集（如已有自研OS），或同时集成工具集+openRuyi镜像。

2.  **突破潜在重量级伙伴（华创微 HC Semi）**：
    *   **机会点**：芯片刚流片成功，正处于生态构建的黄金窗口期，有望定义其官方SDK标准。
    *   **行动建议**：由PLCT实验室或软件所高层牵头接洽，提议打包提供“RuyiSDK工具集 + openRuyi参考系统”的全套生态方案。将RuyiSDK工具集作为其官方开发环境的基础，同时将openRuyi作为其芯片的参考操作系统之一，实现“芯片流片即支持”的生态效果。

3.  **探索生态互补型伙伴（南京沁恒微电子 WCH）**：
    *   **现状**：WCH工具链自闭环，在MCU领域拥有海量开发者。
    *   **行动建议**：此场景更适合RuyiSDK工具集单点切入。
        *   **轻量级合作**：将RuyiSDK包管理器作为获取WCH官方SDK、数据手册和例程的可选渠道，方便Linux/macOS开发者。
        *   **场景化合作**：针对其**CH32H417**新品，联合推出“RISC-V异构计算参考设计”。RuyiSDK工具集负责Linux端开发环境（如Milk-V Jupiter板），CH32H417负责MCU端实时控制，通过USB3.0协同工作。此场景中，openRuyI可作为Linux端的推荐系统之一。

#### 三、生态协同共建：联合操作系统与软件厂商，构建完整闭环

此部分是openRuyi接棒后需要重点梳理的领域，需明确RuyiSDK工具集与各OS厂商的合作边界。

1.  **操作系统厂商（deepin / openKylin / OpenCloudOS / 苦芽科技 / 秦派软件 / 英麒智能）**：
    *   **2026年策略**：从“入驻仪式”转向“产品化落地”，但需明确分工：
        *   **RuyiSDK工具集的角色**：为各OS厂商提供统一的**开发环境构建工具**。各OS厂商在适配新开发板时，可使用RuyiSDK工具集来快速搭建交叉编译环境、生成基础根文件系统等，提升开发效率。
        *   **openRuyi的角色**：作为众多RISC-V操作系统中的一员，与其他OS厂商是**生态伙伴**关系，而非替代关系。openRuyi将专注于提供“开箱即用”的通用系统镜像，与各OS厂商的特色系统形成互补。
        *   **合作落地**：在2026年新开发板的官方推荐操作系统列表中，可同时列出deepin、openKylin、openRuyi等多个选项，由开发者根据需求选择。RuyiSDK工具集负责确保这些系统都能通过`ruyi`命令方便地刷写到开发板上。

2.  **自有操作系统伙伴（Bianbu OS / openEuler RISC-V）**：
    *   **策略**：这些系统与软件所/PLCT有深厚渊源，是天然的深度合作伙伴。
    *   **行动建议**：
        *   **Bianbu OS (进迭时空)**：openRuyi团队可与Bianbu OS团队探讨技术协同，如共享内核优化、共同维护软件包等。RuyiSDK工具集则作为双方共同的开发环境基础。
        *   **openEuler RISC-V**：延续过往合作，RuyiSDK工具集保持对openEuler RISC-V的全面支持。openRuyI可与openEuler RISC-V形成差异化定位，例如openRuyI更侧重于桌面和通用开发场景。

#### 需要内部协同的关键事项

基于上述调整，建议内部明确以下事项：

1.  **分工边界文档化**：由RuyiSDK工具集团队和openRuyi团队共同起草一份《RuyiSDK工具集与openRuyI分工协作白皮书》，明确：
    *   各自的产品定位、技术边界、发布节奏
    *   对外沟通时的统一话术
    *   面向合作伙伴时的协同流程（如：伙伴提出适配需求后，如何分流到工具集团队或openRuyi团队）

2.  **建立跨组协同机制**：
    *   设立双周或月度联席会议，同步项目进展、伙伴动态、技术规划
    *   建立共享的“伙伴需求追踪看板”，确保跨组需求不遗漏、不重复

3.  **统一对外品牌形象**：
    *   在官网、公众号等渠道，明确展示RuyiSDK工具集与openRuyI的关联与分工（例如：RuyiSDK工具集是“开发环境”，openRuyI是“操作系统之一”）
    *   联合参加RISC-V相关会议、展览时，统一展台设计和宣传口径
